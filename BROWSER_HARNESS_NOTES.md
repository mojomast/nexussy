# Browser Harness Feasibility Notes

Phase 0 source reviewed: `https://github.com/browser-use/browser-harness` README, `install.md`, `SKILL.md`, `pyproject.toml`, and `src/browser_harness/`.

## Summary

`browser-harness` is feasible for nexussy only as an optional post-develop browser validation adapter. It should never be required for the existing six-stage pipeline, CI, or fresh installs.

## Packaging

- `browser-harness` is a Python package and CLI, not a Node package or standalone binary.
- The CLI entrypoint is `browser-harness = browser_harness.run:main`.
- It requires Python `>=3.11` and package dependencies `cdp-use`, `fetch-use`, `pillow`, and `websockets`.
- The recommended install is a separate editable tool install such as `uv tool install -e .` from a durable browser-harness checkout.
- Nexussy should not add these packages to required dependencies. Integration should discover an existing `browser-harness` command or configured command path.

## Browser Requirement

- A CDP-compatible browser is required for real browser validation.
- Local mode attaches to an already running Chrome, Chromium, Edge, Brave, Arc, or Comet profile through Chrome DevTools Protocol.
- Remote mode can attach to a Browser Use cloud browser when `BROWSER_USE_API_KEY` is configured.
- Local discovery checks profile `DevToolsActivePort` files, `BU_CDP_WS`, `BU_CDP_URL`, and local `/json/version` probes on ports `9222` and `9223`.
- `browser-harness --doctor` is read-only and exits success only when Chrome and the daemon are healthy. It reports Chrome running, daemon alive, active browser connections, optional `profile-use`, and optional `BROWSER_USE_API_KEY`.

## Detection And Fallback

Recommended nexussy behavior:

- Default `stages.validate_browser.enabled` to `false`.
- If disabled, do not alter automatic behavior or the six core stages.
- If enabled but `browser-harness` is not found, mark browser validation as skipped with a clear diagnostic.
- If enabled but doctor/health checks fail, mark browser validation as skipped or failed based on config, with no attempt to launch Chrome interactively in CI.
- Treat `BROWSER_USE_API_KEY` as optional and never print it.
- Keep CI tests fake-only by mocking the command runner and CDP/browser output.

## Minimal Validation Session

A minimal CLI session can create a new tab, load a local or served URL, wait for document load, collect page metadata, drain CDP events, and run JavaScript to collect browser-visible errors.

Example shape, to be generated by nexussy only when the feature is enabled:

```bash
browser-harness -c '
new_tab("http://127.0.0.1:PORT/")
wait_for_load()
errors = js("""
return {
  title: document.title,
  url: location.href,
  readyState: document.readyState,
  bodyText: document.body ? document.body.innerText.slice(0, 2000) : ""
}
""")
events = drain_events()
print({"page": errors, "events": events})
'
```

For console errors specifically, nexussy may prefer a small injected listener before navigation or a direct CDP event drain if `Runtime.consoleAPICalled` / exception events are available in the drained event buffer. The adapter should normalize findings into a small JSON artifact rather than exposing raw browser logs unbounded.

## Optionality

- The integration can be skipped gracefully.
- Browser setup is interactive by design for local Chrome remote-debugging permissions, so nexussy must not require setup during normal pipeline runs.
- Headless/server use requires either a preconfigured CDP endpoint through `BU_CDP_WS` / `BU_CDP_URL` or Browser Use cloud credentials.
- A safe nexussy implementation should expose explicit opt-in config, bounded command timeout, bounded stdout/stderr capture, and a deterministic skipped result when prerequisites are missing.

## Implementation Implications For Sprint 5

- Add a separate `validate_browser` stage only when explicitly enabled.
- Place browser validation after `develop` so it can inspect built output from changed files.
- Preserve existing stage order and behavior when disabled.
- Do not add `browser-harness` to core required dependencies.
- Add tests with fake command execution and no real Chrome dependency.
